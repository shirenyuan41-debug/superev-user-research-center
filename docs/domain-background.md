# 背景知识文档

## 1. 文档目的

本文档用于补齐项目背后的业务背景、研究语境和系统上下文，帮助后来接手这个项目的人快速理解：

- 这个系统服务谁
- 研究的是哪类业务
- 为什么需要这些模块
- 为什么当前会优先建设 Prompt 治理、反馈沉淀和数据持久化

## 2. 业务主体：超级电动

根据项目内置业务设定，超级电动是一家新能源电车订阅平台。它的核心不是一次性卖车，而是提供更灵活的“用车权”方案。

项目里反复出现的高频词包括：

- 新能源电车
- 订阅
- 换新
- 买断
- 灵活租期
- 智驾
- 贬值焦虑

这说明业务真正解决的问题不是“能不能有车开”，而是“如何更灵活、更省心、更少承担技术迭代和资产折旧风险”。

## 3. 两条核心业务线

## 3.1 超级订阅

这是偏长期的订阅方案，特点包括：

- 按月付费
- 长期拥有新车使用权
- 到期可换新车 / 买断 / 续约
- 常见周期为 6、12、24 个月

用户为什么会选择它：

- 不想承担电车快速贬值
- 想持续开较新的车型
- 不想处理二手车转卖
- 更看重总拥有成本和省心体验

## 3.2 灵活订阅

这是偏短期和高灵活的方案，特点包括：

- 租期从 1 周到 12 个月
- 可以随时换车或退订
- 更适合短期尝鲜、临时用车、阶段性跨城生活

用户为什么会选择它：

- 临时用车
- 体验新车型
- 短期跨城
- 不想被长期绑定

## 4. 目标用户特征

当前项目隐含的核心用户特征大致是：

- 25-40 岁城市中产
- 对智驾、车机、车型更新敏感
- 会在买断和订阅之间做理性比较
- 有明显的风险规避心理
- 不只在意价格，也在意规则是否透明、服务是否省心

因此，这类用户不是“只想更便宜”，而是“想降低长期不确定性”。

## 5. 为什么这个业务需要研究系统

新能源订阅业务的决策链远比普通电商复杂，通常同时包含：

- 功能判断：车够不够好开、配置够不够新
- 财务判断：买断和订阅谁更划算
- 信任判断：规则透明不透明、权益会不会缩水
- 时间判断：交付和换车是否顺畅
- 身份判断：这种用车方式是不是符合自我认同

这决定了项目不能只做简单台账，而需要同时具备：

- 访谈分析
- 画像归纳
- 旅程梳理
- 反馈归因
- Prompt 治理

## 6. 当前系统为什么优先做“用户研究”

虽然导航里已经有多个研究模块，但当前最先做实的是“用户研究”，原因很直接：

- 文档材料最明确
- 分析结构最清晰
- 反馈最容易回流
- 最适合作为 Prompt 迭代起点

因此当前代码里的真实能力主要集中在：

- 用户研究文档上传
- 文本解析
- 结构化分析
- 用户画像结果消费
- 使用反馈
- Todo 跟进

## 7. 业务中的高频认知冲突

## 7.1 “买更划算” vs “订更省心”

很多用户一开始会按月租金直觉觉得贵，但真正拉长到 1-3 年后，又会受到贬值、换车、卖车麻烦、技术迭代等因素影响。

所以研究里不能只问“贵不贵”，还要问：

- 用户怎么算账
- 用户在哪一步开始犹豫
- 用户究竟在担心现金流、总成本，还是后续麻烦

## 7.2 “想换新” vs “怕规则不透明”

用户喜欢“年年开新车”的叙事，但一旦换车规则、补差价逻辑、车况定义说不清，问题就会立刻从“产品优势”变成“信任风险”。

这意味着：

- 权益透明度本身就是核心体验
- 销售口径、官网文案、产品界面必须尽量一致

## 7.3 “电车升级快” vs “买了就过时”

当前项目首页和画像信息里都在强调一个趋势：

- 焦虑正在从“电池衰减”转向“智驾过时”

这意味着订阅并不是“买不起车时的替代”，而是“应对技术快速迭代的一种主动选择”。

## 7.4 “我听懂了” vs “我其实没搞明白”

很多用户在沟通中表面认可规则，实际上并没有真正理解。这也是为什么当前 Prompt 特别强调：

- 潜台词识别
- 模糊表达识别
- 人物角色识别
- 冲突信息识别

## 8. 为什么当前项目强调定性研究

在这个业务里，很多关键问题不是靠纯量化表就能解释清楚。例如：

- 为什么用户说价格贵，真正卡点却是押金
- 为什么用户说想换新，本质是在回避残值风险
- 为什么用户说功能没问题，最后还是没有下订

这类问题更适合通过以下方法去挖：

- 主题分析
- 编码
- 心智模型
- JTBD
- 情感分析
- 溯因推理

## 9. 关键术语说明

## 9.1 JTBD

Jobs To Be Done，强调用户“雇佣”产品去完成什么任务。

在这个项目里，用户的任务通常不是单纯“获得一辆车”，而是：

- 灵活拥有一辆体面的新能源车
- 避免技术过时和残值损失
- 降低不同生活阶段之间的换车摩擦

## 9.2 洞察

洞察不是原话，也不是摘要，而是：

- 从现象出发
- 解释背后机制
- 对业务决策有意义的判断

## 9.3 用户旅程

用户旅程是从“产生需求”到“下决策或放弃”的过程。当前项目常见阶段包括：

- 产生需求
- 搜索对比
- 首次咨询
- 关键顾虑
- 下决策

## 9.4 画像

画像不是标签堆砌，而是帮助团队理解：

- 这类人是谁
- 关心什么
- 害怕什么
- 为什么这样做决策

## 9.5 用研反馈

本项目中的“反馈”不是 C 端用户评价，而是内部使用者对 AI 分析结果的评价，它的作用是反哺 Prompt 和分析流程。

## 10. 为什么系统要把 Prompt 和反馈做成后端能力

早期原型里，Prompt、反馈、Todo 和文档都可以先放在浏览器里跑通，但一旦进入多人共用或持续迭代阶段，就会立刻遇到问题：

- 浏览器数据不稳定
- 结果不可追溯
- Prompt 无法版本治理
- 反馈无法跨人共享
- 旧数据难迁移

因此当前系统优先做了这几件基础设施：

- MySQL 持久化
- Prompt 模块和版本表
- 反馈表
- Todo 表
- 文档表
- 会话表

这不是“技术炫技”，而是为了让研究流程真正能累计资产。

## 11. 当前多模块设计背后的业务意义

虽然很多模块还没接真实模型，但模块划分已经表达了团队的长期方向：

### 用户研究

聚焦用户访谈、需求和心智

### 销转研究

聚焦成交阻力、话术机会和转化分析

### 舆情研究

聚焦社媒风险、话题和情绪变化

### 行业研究

聚焦行业趋势、竞品和市场变化

### 员工研究

聚焦组织反馈、团队状态和流程问题

这意味着后续平台不是单点工具，而是一个统一的研究运营底座。

## 12. 建议团队共享的判断原则

- 不要把“用户没听懂”简单归因为用户问题
- 不要把“价格贵”当成默认根因
- 不要把“建议加强沟通”当成合格行动建议
- 研究输出必须尽量能转成可执行动作
- Prompt 优化要以反馈样本为依据，而不是只靠个人感觉
